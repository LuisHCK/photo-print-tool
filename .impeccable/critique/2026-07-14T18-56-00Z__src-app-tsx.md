## Anti-Patterns Verdict

**LLM assessment:** The UI avoids every DON'T item. No gradient text, glassmorphism, numbered markers, decorative grids, sketchy illustrations, stripe backgrounds, or over-rounded corners. It's recognizably a shadcn/ui+Tailwind product tool with a warm terracotta palette — not a ChatGPT-generated landing page.

**Deterministic scan:** The detector ran clean (exit 0, zero findings) across all 5 source files. No anti-patterns flagged. However, static analysis caught issues the pattern scanner doesn't check for: contrast ratio failures, keyboard accessibility gaps, and dead boilerplate CSS.

## Overall Impression

A solid, functional tool that communicates competence through its three-panel layout and real-time preview warnings. The core value prop — precise print layout without design software — is delivered. The biggest opportunity is the discovery problem: all settings collapsed by default makes the tool feel empty and forces users to hunt for capabilities. The settings section looks like a ghost town on first load. Fix that single thing, and the whole UX lifts a tier.

## What's Working

1. **Real-time preview with actionable warnings** — overflow detection (red) and PPI quality warnings (amber) update live as settings change. This is the tool's best feature: it directly prevents bad prints.

2. **Three-column responsive layout** — 320px | 1fr | 360px grid gives the preview maximum space, with the photos panel and settings panel at fixed widths. Collapses to single column on smaller screens.

3. **Settings profiles with localStorage persistence** — a full save/load/delete system. Unusual depth for a tool this size, genuinely useful for power users doing repeated print jobs.

## Priority Issues

### P0: All settings collapsed by default
`print-settings-panel.tsx:24-29` — every section starts collapsed. User sees a card with title + description and a list of empty accordion headers.
- **Why it matters:** First-timers don't know the tool has capabilities. The right panel looks broken/empty. Power users waste clicks every session re-opening the same sections.
- **Fix:** Open "Layout" by default (`layout: true`). Better: auto-open relevant sections based on state — when photos are loaded, open Size & Spacing.
- **Suggested command:** `onboard`

### P0: No undo or confirm on destructive actions
`removePhoto` (`photos-panel.tsx:89`), `deleteSettingsProfile`, print (`App.tsx:10`). No confirmation for any of these.
- **Why it matters:** Accidental photo removal means full re-upload. Accidental print wastes paper and ink. Zero recovery path erodes trust.
- **Fix:** Add AlertDialog on photo remove and profile delete. Add a print review dialog showing page count, paper type, and PPI summary before `window.print()`.
- **Suggested command:** `harden`

### P1: Grid alignment presents 9 simultaneous options
9 icon buttons in a 3x3 grid. Icons require hover/tooltip to understand.
- **Why it matters:** Violates minimal-choice cognitive load rule. First-timers won't know what "grid alignment" means. 9 options for a decision that's usually "center."
- **Fix:** Default to "Center" and offer a simpler 5-option selector (corners + center). Relegate full 9-option grid to an "Advanced" toggle.
- **Suggested command:** `distill`

### P1: Muted-foreground contrast fails WCAG AA
`--muted-foreground: #83827d` on `--muted: #ede9de` yields ~3.24:1 contrast ratio — fails the 4.5:1 minimum for normal text. Used in `photos-panel.tsx:66` for 14px photo names and throughout the UI for descriptions and metadata.
- **Why it matters:** Users with low vision or viewing on low-brightness screens will struggle to read secondary text. This is the most common WCAG failure in AI-generated designs.
- **Fix:** Darken `--muted-foreground` to at least ~#6b6a64 (or adjust the muted background to be lighter). Target 4.5:1 on the muted surface.
- **Suggested command:** `audit`

### P1: No upload progress indicator
`handleFileUpload` processes photos async with `Promise.all` on image decoding, but no spinner or loading state exists.
- **Why it matters:** With 10+ large photos, the UI freezes for seconds. Users see nothing and may re-click or navigate away.
- **Fix:** Add `isUploading` state, show a spinner with photo count in the Photos panel during processing.
- **Suggested command:** `polish`

### P2: Preview slot not keyboard accessible
`preview-panel.tsx:91-93` — a `<div>` with `onClick` but no `role="button"`, no `tabIndex`, no `onKeyDown`.
- **Why it matters:** Keyboard-only users cannot select/activate photo slots. Screen reader users will not discover them.
- **Fix:** Add `role="button"`, `tabIndex={0}`, and `onKeyDown` handler.
- **Suggested command:** `audit`

### P2: "Fill" vs "Fit" unexplained
`print-settings-selected-photo-category.tsx` — two cryptic mode names with zero help text.
- **Why it matters:** Casual users don't know "fill" means cropping and "fit" means letterboxing. Wrong choice = ruined print.
- **Fix:** Add a one-liner below the select: "Fill: photo fills entire cell (may crop). Fit: photo fits inside cell (may leave borders)."
- **Suggested command:** `clarify`

### P2: Dead boilerplate CSS conflicts with layout
`src/App.css` contains unused Vite boilerplate classes and a `#root` rule (`max-width: 1280px; margin: 0 auto; padding: 2rem; text-align: center;`) that conflicts with the full-width grid layout.
- **Why it matters:** The `#root` padding and text-align may subtly interfere with the grid layout, especially on mid-size viewports.
- **Fix:** Remove `App.css` entirely or strip all content — it's pure leftover Vite scaffold.
- **Suggested command:** `polish`

## Persona Red Flags

**Alex (Power User):**
- All sections collapsed every session — no persistence of expanded state across sessions. Must manually re-open the same sections.
- Profiles feature buried 3 levels deep (Settings → Profiles accordion → Manage → select/load) despite being the most power-user feature.
- No keyboard shortcuts: Ctrl+P doesn't map to the app's print. No `←`/`→` for preview pages.
- No batch operations: can only toggle photos one at a time. No "select all" or "invert."

**Jordan (First-Timer):**
- Lands on three-panel layout with no clear starting point. The photos panel says "Import photos" (a label, not a button) — the actual interaction is the file input below.
- All settings hidden behind collapsed accordions. Jordan won't know what "Layout & copies" or "Size & spacing" contain and may never expand them.
- "Fill (crop)" vs "Fit (contain)" — no explanation, no visual preview. Jordan will guess.
- PPI warning "~72 PPI" — Jordan has no idea what PPI means or what number is good.

**Sam (Accessibility):**
- PPI/overflow warnings are color-only (`text-amber-700`, `text-destructive`) — fails WCAG 1.4.1 Use of Color.
- Muted-foreground (#83827d) on muted (#ede9de) at ~3.24:1 contrast — fails WCAG AA for normal text.
- Preview slot div has onClick but no keyboard handling or ARIA role.
- Grid of 9 alignment icons with similar arrow shapes may be hard to distinguish for low-vision users.

## Minor Observations

- Theme toggle icon shows *target* state (Sun in dark mode, Moon in light mode) — common pattern but may confuse.
- `photos-panel.tsx:67` — fixed `w-40` (160px) for filename truncation may still break on long filenames.
- `print-settings-panel.tsx:40` — `max-h-screen` + `max-h-[calc(100vh-2rem)]` creates scrollable panel inside scrollable page. Double-scrollbar risk.
- `--font-serif` set to Space Grotesk, which is a sans-serif typeface — naming mismatch if ever relied upon.
- `photos-panel.tsx:38` — fixed `62vh` ScrollArea may overflow on very short viewports (<500px).
- Defaulting units to `cm` may surprise US users expecting inches.

## Questions

1. **What if settings were organized by "what you're doing" rather than "what category it is"?** A user who just uploaded photos wants Size & Spacing, not Layout first. A wizard-like progressive flow (Upload → Configure → Preview → Print) would eliminate the collapsed-sections problem entirely.

2. **What if profiles became the top-level organizational concept?** Switching profiles = switching all settings instantly. That's more powerful than a buried accordion section.

3. **What if the print button opened an app-level Print Review?** Showing page count, PPI summary per page, paper cost estimate — giving users a confident final moment instead of a jarring browser dialog.
